# Reviewing this provider: what to actually read

Most of the repo is generated code or shared template/tooling. Of the hand-written
Go (~800 lines), the provider-specific logic lives in a handful of files. This
guide points a new reviewer at those and says what to skip.

> **Rule of thumb:** the API type is the source of truth; the `zz_generated.*`
> files and `package/crds/*.yaml` are *derived* from it. Review the type, trust
> (don't read) the generated output.

## 1. Start here — the core provider logic

If you only have 20 minutes, these are the review.

| Priority | File | Why it matters |
| -------- | ---- | -------------- |
| 1 | `internal/controller/instance/instance.go` | The `ExternalClient`: `Observe`/`Create`/`Update`/`Delete` — the reconcile contract and the heart of the provider. |
| 1 | `internal/controller/instance/client.go` | HTTP client to the external API: endpoint/auth handling, request/response shaping. |
| 1 | `apis/database/v1alpha1/instance_types.go` | The API surface — `Instance` spec/status (`forProvider` / `atProvider` fields). |
| 2 | `internal/controller/instance/instance_test.go` | How the lifecycle is tested (`httptest`); doubles as executable documentation. |

## 2. Supporting wiring — how it's assembled

Small, mostly hand-edited from the template. Read to see how the pieces connect.

| File | Why |
| ---- | --- |
| `cmd/provider/main.go` | Entrypoint: flags, `SetupGated` controller wiring. |
| `apis/v1alpha1/types.go` | `ProviderConfig` credential types. |
| `internal/controller/config/config.go` | `ProviderConfig` controller setup. |
| `apis/v1alpha1/register.go` | `ProviderConfig` scheme registration. |
| `internal/controller/learn.go` | Which controllers get registered. |
| `apis/learn.go`, `apis/database/database.go` | API group registration. |

Helps a reviewer actually run the thing.

| Path | What |

## 4. Narrative context (recommended on-ramp)

## Safe to skip — generated or untouched boilerplate

Don't spend review time here.

| What | Why skip |
| ---- | -------- |
| `apis/**/zz_generated.*.go` (7 files) | Code-generated (`deepcopy`, `managed`, `pc`/`pcu`); "DO NOT EDIT", regenerated by `make generate`. |
| `package/crds/*.yaml` | Generated from `instance_types.go` and the `ProviderConfig` types. |
| `apis/**/doc.go`, `groupversion_info.go` | Tiny codegen markers / boilerplate. |
| `apis/generate.go` | Just `go:generate` directives. |
| `build/` | The `crossplane/build` submodule (shared tooling). |
| `Makefile` | Template; only org/name edits. |
| `_output/` | Build artifacts (gitignored). |
| `cluster/`, `hack/boilerplate.go.txt`, `renovate.json` | Template scaffolding. |
| `LICENSE`, `DCO`, `OWNERS.md`, `CODE_OF_CONDUCT.md`, `PROVIDER_CHECKLIST.md` | Licensing / governance. |

## A suggested 30-minute path

1. `apis/database/v1alpha1/instance_types.go` — what an `Instance` *is*.
2. `internal/controller/instance/client.go` — how it talks to the API.
3. `internal/controller/instance/instance.go` — the reconcile lifecycle.
4. `internal/controller/instance/instance_test.go` — confirm behaviour.
5. Skim `hack/LEARNING-JOURNAL.md` for the *why*, then run
   `hack/TESTING-OUT-OF-CLUSTER.md` to see it work end-to-end.
